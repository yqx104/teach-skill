# MISSION.md Format

`MISSION.md` 位于工作区根目录。它记录用户学习该主题的**原因**。每一个教学决策——接下来教什么、展示哪些资源、设计哪些练习——都应该能追溯到这个文档。

## 模板

```md
# 使命：{主题}

## 为什么
{1-3 句话。用户正在追求的具体、现实目标。掌握这项技能后，用户的生活或工作会发生什么变化？避免使用“为了理解 X”这类抽象表述——要继续追问其背后的真实结果。}

## 成功的样子
- {用户将能够完成的一件具体、可观察的事情}
- {另一件具体的事情}
- {……}

## 约束
- {时间、预算、已有承诺、学习偏好，以及任何会限制学习方式的因素}

## 不在范围内
- {用户明确表示当前不想追逐的相邻主题——用于保护用户的最近发展区}
```

## 规则

* **一个工作区只对应一个使命。** 如果用户想学习两个互不相关的东西，那就应该创建两个工作区。
* **具体优先于抽象。** “在 10 月前跑完半程马拉松”比“变得更健康”更好。“向团队交付一个 Rust CLI 工具”比“学习 Rust”更好。
* **对模糊目标提出质疑。** 如果用户说不清为什么要学，就先采访他们，再写任何内容。一个糟糕的使命比没有使命更糟。
* **当现实发生变化时进行修订。** 使命会变化。当用户的目标改变时，更新这个文件——不要让过时的使命继续指导未来的学习。
* **保持简短。** 如果 `MISSION.md` 超过一屏，它就不再是指南针，而变成了计划。
